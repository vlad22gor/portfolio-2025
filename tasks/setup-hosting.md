# Portfolio Setup Decision: Astro + GitHub Pages (Cloudflare Trigger)

## 1) Контекст и цели миграции

Цель: перенести портфолио с Framer в кодовую базу, которую удобно поддерживать в Codex App через Git-процесс.

Что важно:
- Простая эксплуатация и деплой.
- Контроль над кодом и контентом без зависимости от Framer runtime.
- Сохранение текущего URL-контракта: `/`, `/cases`, `/gallery`, `/fora`, `/kissa`.
- Mobile-first адаптив вместо desktop-only поведения.

## 2) Аудит текущего сайта (vladhorovyy.com)

### Наблюдения по структуре
- Сайт уже многостраничный: `/`, `/cases`, `/gallery`, `/fora`, `/kissa`.
- Контентный тип: персональный лендинг + кейсы + визуальная галерея.
- Back-end логика отсутствует: форм и серверных обработчиков нет.

### Наблюдения по реализации
- Реализация на Framer runtime (встроенный CSS/JS и Framer-specific классы/скрипты).
- HTML страниц тяжёлый (сотни KB на страницу), много медиа-ассетов.
- Текущая версия использует desktop-first поведение с ограничением для меньших экранов.

### Вывод аудита
Перенос "как есть" (копирование Framer HTML/CSS/JS) технически возможен, но создаёт высокий техдолг. Правильнее перенести контент и собрать компонентную структуру заново на SSG.

## 3) Сравнение хостингов: GitHub Pages vs Cloudflare Pages

## GitHub Pages

Плюсы:
- Нативно встраивается в GitHub workflow.
- Простой CI/CD через GitHub Actions.
- Достаточно для статического портфолио.
- Прозрачный процесс для личного сайта и PR-обновлений.

Риски:
- Soft-лимиты по билдам и трафику (при росте медиа-нагрузки можно упереться).
- Нет edge/runtime преимуществ из коробки.
- Для сложных динамических сценариев экосистема менее удобна, чем у Cloudflare.

## Cloudflare Pages

Плюсы:
- Сильный edge/CDN-профиль и запас по статике.
- Хороший путь к Functions/Workers при будущей динамике.
- Гибкий контроль доставки и кеширования.

Риски:
- Чуть больше операционной сложности на старте при твоём GitHub-first сценарии.
- Добавляет ещё одного инфраструктурного вендора в обязательный контур.

## 4) Финальное решение и условия пересмотра

## Решение сейчас
Выбираем **Astro + Tailwind + GitHub Pages + GitHub Actions**.

Почему это оптимально под текущий кейс:
- Сайт статический.
- Приоритет: простота и скорость запуска через Git.
- Можно сохранить URL-контракт без SSR.
- Компонентная архитектура снижает стоимость поддержки по сравнению с Framer runtime export.

## Когда пересматривать и переходить на Cloudflare Pages
- Стабильный рост трафика и медиа-нагрузки, бьющий в лимиты GitHub Pages.
- Появление требований к edge-логике, functions или более гибкому кешированию.
- Необходимость строгого перф-контроля на глобальной аудитории.

## 5) Пошаговый migration plan + release checklist

## План миграции
1. Создать SSG-проект на Astro с Tailwind.
2. Завести типизированный контракт `Case` и перенести контент кейсов в данные.
3. Реализовать страницы: `/`, `/cases`, `/gallery`, `/fora`, `/kissa`.
4. Добавить mobile-first брейкпоинты: `390`, `768`, `1024`, `1440+`.
5. Перенести медиа локально (или в object storage + CDN), исключив зависимость от Framer runtime.
6. Настроить GitHub Actions deploy workflow и `CNAME`.

## Release checklist
- [ ] Все маршруты открываются и соответствуют ожидаемым URL.
- [ ] Навигация между кейсами (prev/next) работает.
- [ ] Нет desktop-only fallback на мобильных.
- [ ] Проверен Lighthouse: LCP/CLS/INP на ключевых страницах.
- [ ] Медиа оптимизированы (WebP/AVIF, lazy loading, размерные бюджеты).
- [ ] GitHub Actions deploy проходит без ошибок.
- [ ] HTTPS и custom domain корректно работают.
- [ ] Базовая доступность: keyboard navigation, alt-тексты, контраст.

## Риски и митигации
- Риск: долгий путь до 1:1 повторения Framer-визуала.
  Митигация: сначала информационный каркас и UX, затем микродетали и анимация.

- Риск: разрастание веса ассетов.
  Митигация: заранее задать budget и добавить оптимизацию в контент-процесс.

- Риск: ограничение GitHub Pages при росте.
  Митигация: заранее описанный switch-path на Cloudflare Pages без смены контентной модели.

## Принятые допущения
- Контент обновляется через Git PR, без CMS на старте.
- Серверные функции не нужны в горизонте 3-6 месяцев.
- Хостинг-приоритет: простота и предсказуемость эксплуатации.
